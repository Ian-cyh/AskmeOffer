# 需求分析文档

> 项目：AskmeOffer — 保研全流程 AI 助手  
> 更新时间：2026-05-24

---

## 一、目标用户画像

**核心用户**：正在准备保研（推免）的理工科本科生，大三为主。

**典型用户特征**：

| 维度 | 描述 |
|------|------|
| 阶段 | 大三下 ~ 大四上，距夏令营/预推免 2–8 周 |
| 背景 | 理工科（CS、EE、自动化、数学等），成绩在保研线附近或以上 |
| 资源 | 缺乏成功保研的学长指导，信息获取碎片化 |
| 焦虑 | 材料准备、专业课复习、面试练习多线并行，时间不够用 |
| 行为 | 独自准备为主，偶尔在保研论坛/群聊获取零散信息 |

**暂不覆盖**：

- 考研学生（备考逻辑完全不同）
- 出国申请（材料体系不同）
- 文科/商科保研（考核形式差异大）

---

## 二、核心痛点

### 痛点 1：专业课考核准备无方向
- 各院校考核科目差异大，不知道重点在哪
- 缺乏目标院校的针对性模拟测试

### 痛点 2：上机代码考核缺乏针对性训练
- 院校机试风格不同（算法竞赛 vs 工程应用）
- LeetCode 刷了很多但不知道目标院校考什么

### 痛点 3：面试准备碎片化
- **项目经历讲不透**：做过的项目说不清技术细节和个人贡献
- **压力面应对差**：追问时大脑空白，越紧张越答不好
- **面试重点把握不准**：不知道面试官最关注什么

### 痛点 4：导师/院校信息获取困难
- 导师方向、代表作、招生偏好散落各处
- 没有统一的管理和追踪工具

### 痛点 5：个人材料管理繁琐
- 简历、个人陈述、推荐信等材料需要针对不同院校微调
- 修改个人信息后所有材料需要手动同步，版本混乱

## 三、产品模块设计

### 模块 1：个人信息模块 ⭐ MVP 核心

**子功能**：

| 子功能 | 说明 |
|--------|------|
| 结构化信息录入 | 基本信息、成绩、课程、竞赛、论文、荣誉、项目、学生工作、推荐信 |
| AI 简历生成 | 基于录入信息自动生成结构化简历，信息修改后自动更新 |
| AI 个人陈述生成 | 基于信息 + 目标院校生成个人陈述，支持微调 |
| 目标院校管理 | 院校/院系信息、导师方向、代表作、联系进度追踪 |
| 个人反馈中心 | 汇总面试/专业课/机考历史结果，生成能力画像 |

### 模块 2：面试模块

| 功能 | 说明 |
|------|------|
| 模拟面试 | 基于个人信息（简历）进行 AI 模拟面试 |
| 难度调节 | 简单 / 标准 / 压力面 |
| 追问机制 | 根据回答在薄弱点深挖 ≥ 2 层 |
| 结构化反馈 | 每轮结束后：优点 / 问题 / 参考要点 |

### 模块 3：专业课模块

| 功能 | 说明 |
|------|------|
| 针对性复习 | 根据目标院校生成专业课复习路径 |
| 模拟问答 | AI 扮演考官进行知识点口试 |
| 复习反馈 | 识别薄弱知识点，推荐复习资源 |

### 模块 4：机试模块

| 功能 | 说明 |
|------|------|
| 风格匹配 | 根据院校机试特点推荐练习方向 |
| 代码实战 | 模拟机试环境练习 |
| 薄弱分析 | 按知识点维度分析弱项 |

## 四、开发优先级（16小时策略）

| 优先级 | 模块 | 实现深度 |
|--------|------|---------|
| P0 | 个人信息模块 | 完整实现（录入 + AI 生成 + 院校管理） |
| P1 | 面试模块 | 可交互 Demo（对话式面试 + 反馈） |
| P2 | 专业课模块 | 前端框架 + 占位页面 |
| P2 | 机试模块 | 前端框架 + 占位页面 |

## 五、核心用户旅程

```
[落地页] 欢迎 → 快速了解产品价值
    ↓
[个人信息] 结构化录入基本信息、成绩、成果、经历
    ↓
[材料生成] AI 一键生成简历 & 个人陈述
    ↓
[目标院校] 添加目标院校 & 导师信息
    ↓
[模拟面试] 基于简历进行 AI 面试练习（可选难度）
    ↓
[反馈中心] 查看能力画像、历史表现、薄弱点
```

## 六、差异化定位

> 相比于直接使用 ChatGPT：

1. **数据持久化**：个人信息一次录入，所有模块复用，不用每次重新描述背景
2. **材料联动**：修改信息后简历、个人陈述自动同步更新
3. **场景专业化**：针对保研流程设计，不是通用对话
4. **结构化反馈**：面试/专业课/机考的反馈格式固定、可追踪、可对比
5. **院校信息管理**：整合导师调研，追踪联系进度

---

## 七、技术约束与假设

- LLM API streaming 减少延迟感知
- 无登录体系，session / localStorage 存储用户数据
- 单用户场景，暂不考虑并发
- 部署：前端 Vercel / 后端阿里云 ECS 单机

---

## 八、刻意不做（Won't Have）

| 功能 | 不做原因 |
|------|----------|
| 语音面试 | 实现复杂，文字反馈对复盘更友好 |
| 完整账号系统 | 16小时不够，用 localStorage 替代 |
| 推荐信自动生成 | 涉及推荐人隐私和风格，AI 难以把控 |
| 排行榜/社交 | 非核心痛点 |
| 付费功能 | Demo 阶段不做 |
